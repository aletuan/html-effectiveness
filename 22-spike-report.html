How should we deliver real-time updates to the activity feed?

Ready for decision
owner M. Okafor started Apr 3 decision needed by Apr 12 spec PLATFORM-RFC-014

Context

The activity feed currently polls /api/feed every 5 seconds from every open tab. Three things prompted this spike:

  • Polling now consumes ~22% of API CPU at peak. Capacity planning for Q3 looks bad if we add another polling client.
  • 14 customer complaints last quarter mention “the feed feels stale” or “updates take forever”.
  • Mobile clients drain noticeably more battery than competitors. Profile traces point at the 5s wake interval.

Constraints

  • Ship within Q2 (10 weeks).
  • Must support 50k concurrent connections at peak.
  • No new ops team — current platform team owns infra.
  • Reverse proxy is nginx, no L7 protocol awareness.
  • Mobile clients: iOS Safari 16+ and Android Chrome — both modern enough for SSE.

Options

ATune the existing polling

Keep HTTP polling, but cut the interval where it’s wasteful and add long-poll fallback for active tabs.

Recommendation

Ship Option B — SSE

Polling has hit its ceiling for our scale — even after tuning, we'd be back here in twelve months. WebSockets is the right long-term answer if we add chat or multiplayer, but the activity feed alone doesn't justify 8 weeks and the operational surface area.

SSE is the smallest change that actually solves the user-visible problem (perceived lag), cuts API CPU by ~70%, and leaves 7 weeks of Q2 for follow-on work. If product later commits to chat, WebSockets becomes a Q3 RFC with the activity feed already migrated to a push model.

What changes if rejected: ship Option A as a 1-week stopgap, but file the Q3 RFC immediately — we should not enter Q4 still polling.

Open questions

  • Does iOS Safari keep an SSE stream alive when the app is backgrounded on cellular? The desktop spec says yes; mobile behavior is undocumented and the two threads on Apple's forum disagree. Next: 24-hour real-device test on iPhone 13 + iPhone 15, both carriers. Owner: M.O., due Apr 10.
  • Does our nginx ip_hash sticky-session config survive the new pod autoscaler? The autoscaler can recycle pods underneath a long-lived connection; we need confirmation the proxy will reconnect cleanly. Next: pair with Infra (Devon P.) on staging during the Apr 8 office hour.
  • Is product planning to reuse the push channel for notifications? If yes, capacity sizing doubles and we need to add per-user fan-out. If no, the activity-feed-only channel is much simpler. Next: 15-min sync with Sara (PM) before Apr 11.
