Design decisions

Choices made where the spec was ambiguous or silent. Default to existing patterns in the codebase; flag here so you can override.

Decision
Reuse the existing Markdown renderer instead of pulling in mdast-util-from-markdown
SPEC §3.2 (formatting) src/comments/Body.tsx:18
The spec says comments support “basic Markdown” but doesn’t enumerate which features. The codebase already ships a hardened renderer used by task descriptions (links, bold, lists, fenced code) so I wired comments through that. Adds zero new dependencies and matches the rendering of task descriptions for free.
If you wanted GFM tables or task lists The shared renderer doesn’t do those. Adding them is a separate change and should land in the renderer module, not the comments code.
Decision
Cache thread_head_count on the task row, not in a separate denorm table
SPEC §4.1 (board view perf) db/migrations/0142_thread_head.sql
Spec calls for board-view perf parity (<120ms p95) but doesn’t prescribe a strategy. A column on tasks is the smallest change that meets the bar — board queries already select the row, so the count comes along for free. Updates happen inside the comment create/delete transaction.
Decision
Server-rendered relative timestamps, absolute on hover
SPEC §5.4 (“show relative time”) src/comments/Timestamp.tsx
“Relative time” is underspecified — client-side ticking is the obvious option but it’s an accessibility nightmare and re-render churn. Went with server-rendered strings (“3 min ago”, “yesterday”) and a <time title="..."> tooltip for the exact value. Matches how task activity rows already work.

Deviations

Places I built something different from what the spec said. Each has a reason — push back if any of these don’t match what you wanted.

Deviation
Modal editor on Create, not inline as the spec mock shows
SPEC §2.3 + Figma frame 4 src/comments/CommentComposer.tsx
The mock shows an inline composer that expands beneath the last comment. Keyboard focus and screen-reader semantics for “area that grows inline” are not addressed in the spec — and we got bitten by exactly this on the bulk-edit toolbar.
Built a modal with a single <textarea> instead. Same affordance, much simpler focus handling. If you want the inline version, the modal can be swapped without touching the data layer.
Deviation
No new endpoint — comments returned via GET /tasks/:id?include=thread
SPEC §6.1 (API) src/server/routes/tasks.ts:88
Spec §6.1 lists GET /tasks/:id/comments as a new endpoint. But the board view already fetches the task, and a second round-trip per card would blow the perf budget. Added an include=thread query param to the existing endpoint, gated behind the same authz check. Adds thread only when requested, so the bare GET /tasks/:id response shape is unchanged.

Tradeoffs

Choices with a real alternative. Recording the loser and the reason so we can revisit if conditions change.

Tradeoff
Optimistic UI with rollback, not server-confirm
perceived latency vs. consistency src/comments/store.ts:104
Comments appear immediately on the sender’s screen with a faint “sending…” opacity, then either confirm or rollback with a toast. The other option — wait for server confirmation — is simpler but adds ~180ms of dead time per comment in the staging numbers.
Considered: server-confirm Cleaner code path (one source of truth) but 180ms feels long for a chat-like interaction. Optimistic with rollback is what task title edits already do.
Tradeoff
Polling for new comments, not SSE
infra readiness src/comments/usePolling.ts
We don’t have a shared SSE/WebSocket layer in this service, and adding one for comments alone is a separate — bigger — project. Kept the polling pattern used elsewhere (12s interval, exponential backoff on error). Drop-in replacement when the real-time layer lands.
Tradeoff
Soft delete (tombstone row), not hard delete
matches existing pattern src/comments/delete.ts
Tasks already tombstone with a deleted_at column for the 30-day undo window. Comments now do the same. Hard delete saves storage but breaks the undo affordance and complicates audit log queries that join through comments.

Open questions

Things I had to assume but want you to confirm before I lock in. Replying inline on the PR is enough — I’ll resolve them from there.

Open
@mentions: workspace members only, or include guests?
SPEC §3.5 (mentions)
Spec says “mentionable users” without defining the set. Right now the dropdown filters to active workspace members; guests aren’t suggested even though they can read the task. If guests should be mentionable, the change is one predicate in useMentionables.ts — small.
My assumption (revertible) Members only. Guests can read the comment but won’t be auto-notified on mention.
Open
Retention when the parent task is archived
not addressed in spec
Archiving a task hides it but keeps the row indefinitely. Comments inherit that today — they live forever alongside the archived task. If there’s a PII / retention policy I should follow (e.g., purge after 12 months of archive), the cron job to enforce it is a separate change.
My assumption (revertible) Indefinite retention, matching task rows.
